# v5.8.2 Automation Workflows

This release adds a workflow automation layer on top of the existing global dataset, dataset pipeline, model assistant, downloader, character-reference, augmentation, export, remote-device, and MCP features.

The intent is to let a user select a dataset or branch, describe the training goal, and let the tool build a reproducible curation plan. The plan can be created from templates, generated by the selected assistant/orchestration model, edited directly as JSON by the user, or refined cooperatively by the user and model.

## Design goals

- Keep the **global original dataset immutable**.
- Put all task-specific edits in **branch sidecars/configs/variants**.
- Make workflows editable by both the user and the model.
- Require explicit approval for expensive, destructive, remote, or external-tool actions.
- Support dry-runs before execution.
- Produce manifest files for each run under `runtime/automation_workflows/runs/`.
- Provide a simple GUI while still exposing the underlying workflow JSON for advanced users.

## API

New API routes are available under `/api/workflows`:

| Route | Purpose |
|---|---|
| `GET /api/workflows/catalog` | List workflow templates and allowed step types. |
| `GET /api/workflows` | List saved workflows. |
| `POST /api/workflows` | Create a workflow from a template. |
| `GET /api/workflows/{workflow_id}` | Load one workflow. |
| `PUT /api/workflows/{workflow_id}` | Save direct JSON edits. |
| `DELETE /api/workflows/{workflow_id}` | Delete a workflow. |
| `POST /api/workflows/plan` | Generate a workflow plan from a user goal. |
| `POST /api/workflows/{workflow_id}/refine` | Refine a workflow from user/model instructions. |
| `POST /api/workflows/{workflow_id}/validate` | Validate step types and approval-gate metadata. |
| `POST /api/workflows/{workflow_id}/dry-run` | Simulate the workflow and write a dry-run manifest. |
| `POST /api/workflows/{workflow_id}/run` | Queue a workflow job. |
| `GET /api/workflows/runs` | List recent run manifests. |

Unsafe or expensive steps preserve `requires_approval: true` in the workflow JSON. The run request must include `approve_unsafe_steps: true` before those steps execute.

## Workflow JSON contract

A saved workflow is ordinary JSON. Example shape:

```json
{
  "id": "wf_character_lora_1234abcd",
  "name": "Character LoRA automation",
  "goal": "Prepare this branch for a character LoRA.",
  "assistant_model": "dataset-assistant",
  "source_dataset_id": 1,
  "branch_name": "character_lora_v1",
  "target_model": "sdxl",
  "adapter_type": "lora",
  "dataset_goal": "character",
  "automation_level": "guided",
  "approval_policy": "unsafe_steps_require_approval",
  "steps": [
    {
      "id": "s01_create_branch",
      "type": "create_branch",
      "enabled": true,
      "requires_approval": false,
      "params": { "branch_name": "character_lora_v1" }
    }
  ]
}
```

This same representation is used by the GUI, direct user edits, and assistant-generated/refined workflows.

## Safety model

The workflow engine is conservative by default:

- dry-runs perform no curation-side effects;
- global originals are not modified;
- branch sidecars/configs are the editable layer;
- media variants are branch-local derived files;
- downloads, external trainer export, dictionary network sync, assistant label refinement, and augmentation creation require explicit approval unless the workflow is intentionally run in an approved automation mode;
- model-generated workflows are validated against the allowed step catalog before execution.
