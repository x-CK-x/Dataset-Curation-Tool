# Automation Workflows and Cooperative Curation

![Assistant and orchestration workflow visual](assets/images/assistant_orchestrator_chat.png)

The **Automation Workflows** tab is the workflow layer for making the tool act more like an end-to-end curation pipeline. A workflow can be created by the user, generated by the selected assistant/orchestration model, edited by both user and model, dry-run, then queued as a job.

The goal is not to remove user control. The goal is to let the user define the outcome, then let the tool handle repetitive curation work: linking originals into a global dataset branch, validating labels, pruning wrong images, planning augmentations, creating branch-local variants, checking readiness, and exporting a training-ready branch.

## Mental model

| Layer | Purpose |
|---|---|
| Global Dataset | Holds unique original media once by SHA-256. |
| Branch | Holds model/task-specific configs, sidecar copies, include/exclude flags, and variants. |
| Dataset Pipeline | Defines target-model rules, LoRA/IC-LoRA/ControlNet/embedding guidance, augmentation policy, regularization policy, and export/handoff logic. |
| Automation Workflow | Chains pipeline/global/assistant/download/reference/export actions into an editable automation plan. |
| Assistant / Orchestrator | Can draft, refine, explain, or help repair workflow JSON. |

## What the tab can do

The tab supports:

- creating workflows from templates;
- selecting source dataset, branch, assistant model, target model, adapter type, dataset goal, and training tool;
- generating a workflow from a plain-language goal;
- optionally asking the selected assistant model to draft or refine the workflow;
- editing the workflow JSON directly;
- validating the workflow;
- dry-running the workflow;
- queueing a workflow job;
- reviewing recent workflow run manifests.

## Current templates

| Template | Use case |
|---|---|
| Full automated dataset curation | General end-to-end branch preparation. |
| Character LoRA / OC identity prep | Character identity datasets with optional reference pruning and identity crops. |
| Style LoRA prep | Style-focused caption/tag cleanup and style/detail variants. |
| Download/query to training branch | Source query/download, global dedupe, then branch prep. |
| Existing branch quality gate + export | Evaluate and export a branch that is already mostly curated. |

| Step type | Purpose | Approval default |
|---|---|---|

| `build_model_prompt` | Build a local/cloud model prompt packet. | Auto-safe |
| `assistant_refine_labels` | Ask the assistant model for label refinement guidance. | Approval |
| `dry_run_label_rules` | Preview branch-sidecar cleanup. | Auto-safe |

## Recommended workflow for a character LoRA

1. Import or download the source images.
2. Register/link them into a global dataset branch.
3. Create or select a character reference profile.
4. Run character-reference ranking to identify likely wrong images.
5. Build target-specific caption/tag rules.
6. Dry-run label cleanup.
7. Review the result.
8. Apply label rules to branch sidecars only.
9. Plan character LoRA augmentations.
10. Create branch-local headshot/upper-body/bucket variants where appropriate.
11. Build a regularization plan.
12. Evaluate readiness.
13. Export the branch for the selected trainer/tool.

## Direct JSON editing

Workflows are saved as JSON under:

```text
runtime/automation_workflows/
```

Run manifests are saved under:

```text
runtime/automation_workflows/runs/
```

The GUI JSON editor exposes the exact workflow representation used by the backend. This makes it possible for:

- the user to directly modify steps;
- a model to emit a workflow;
- the user and model to iteratively refine the same workflow;
- future graph/workflow editors to target the same schema.

## Approval and dry-run behavior

Dry-run mode reports what each step would do without side effects. Unsafe or expensive steps keep `requires_approval: true`. Queueing a workflow with **Approve unsafe steps** disabled stops or skips those steps depending on run settings.

This matters because some actions can start large downloads, alter branch sidecars, create many variants, call external tools, or export large training bundles.

## API routes

| Route | Purpose |
|---|---|
| `GET /api/workflows/catalog` | Get step catalog and templates. |
| `GET /api/workflows` | List saved workflows. |
| `POST /api/workflows` | Create a template workflow. |
| `POST /api/workflows/plan` | Generate a workflow from a goal. |
| `PUT /api/workflows/{workflow_id}` | Save JSON edits. |
| `POST /api/workflows/{workflow_id}/refine` | Refine with instructions/model. |
| `POST /api/workflows/{workflow_id}/dry-run` | Dry-run immediately. |
| `POST /api/workflows/{workflow_id}/run` | Queue workflow job. |
| `GET /api/workflows/runs` | List recent run manifests. |

## Relationship to older tabs

Automation Workflows does not replace the existing tabs. It wraps them.

| Existing tab | How workflows use it |
|---|---|
| Downloads | Optional source acquisition step. |
| Global Dataset | Original dedupe and branch layer. |
| Dataset Pipeline | Training-target rules, augmentation, regularization, export. |
| Character Reference | One/few-shot character pruning. |
| Models / Assistant | Optional model-generated planning and label guidance. |
| Remote Devices | Future fork/join or offloaded step execution. |
| Jobs | Workflow runs appear as queued/running jobs. |

## Practical guidance

Start with dry-run. Then run safe steps. Then approve unsafe steps only after inspecting the workflow JSON and dry-run output. For large datasets, use smaller `max_items` values until the workflow produces the expected branch structure and sidecars.
